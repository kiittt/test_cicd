# 测试策略

## 1. 测试目标

通过日程安排 CRUD 项目验证从需求到交付的完整质量流程，确保核心功能正确、接口稳定、页面流程可用，并通过 CI/CD 自动化执行质量门禁。

## 2. 测试分层

| 层级 | 工具 | 目标 |
| --- | --- | --- |
| 后端单元测试 | Go test | 验证业务规则、参数校验、服务逻辑 |
| 后端接口测试 | Go httptest | 验证 HTTP 状态码、响应结构、错误处理 |
| 前端单元测试 | Vitest | 验证工具函数、表单校验、状态转换 |
| 前端组件测试 | Vitest + Testing Library | 验证表单、列表、空状态、错误提示 |
| API 自动化测试 | Playwright request | 验证完整接口 CRUD 流程 |
| UI E2E 测试 | Playwright | 验证真实用户页面操作流程 |
| 静态检查 | ESLint、Prettier、gofmt、go vet | 保证代码风格和基础质量 |

## 3. 第一阶段测试范围

第 1 天只完成项目初始化和测试规划，不要求实现完整功能测试。需要完成以下准备：

- 明确需求、字段和业务规则。
- 明确接口范围和错误码。
- 建立前端、后端、文档目录。
- 确认可执行的测试分层和工具选型。

## 4. 后续核心测试用例方向

| 类型 | 用例 |
| --- | --- |
| 单元测试 | 标题不能为空 |
| 单元测试 | 开始时间不能晚于结束时间 |
| 单元测试 | 优先级只能使用合法枚举 |
| API 测试 | 创建日程后能查询到 |
| API 测试 | 编辑日程后数据正确更新 |
| API 测试 | 删除日程后详情返回 404 |
| UI 测试 | 用户可以新增日程 |
| UI 测试 | 用户可以编辑日程标题 |
| UI 测试 | 用户可以删除日程 |
| UI 测试 | 用户提交空标题时看到校验提示 |

## 5. 质量门禁

后续 CI/CD 中必须满足：

- 前端 lint 通过。
- 前端单元测试通过。
- 前端 build 通过。
- 后端 gofmt 检查通过。
- 后端 go vet 通过。
- 后端单元测试通过。
- API 自动化测试通过。
- UI 核心流程测试通过。

## 6. 测试数据原则

- 自动化用例不依赖固定数据库数据。
- 每条 API 和 UI 用例尽量独立创建测试数据。
- 用例执行后清理自己创建的数据。
- 测试标题增加唯一标识，避免并发或重复执行时冲突。

## 7. 风险

| 风险 | 应对 |
| --- | --- |
| E2E 测试不稳定 | 使用 `data-testid`，等待明确 UI 状态 |
| 测试数据污染 | 每条用例独立创建和清理 |
| 前后端接口不一致 | 第 3 天固定 API 契约 |
| CI 环境与本地不同 | 在工作流中显式安装依赖和浏览器 |
