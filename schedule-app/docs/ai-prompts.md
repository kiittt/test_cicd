# AI 辅助测试提示词

## 使用原则

- AI 只负责生成候选测试点，不直接决定最终用例。
- 输出必须经过人工评审，确认是否符合当前需求、接口契约和自动化成本。
- 对需求未明确的规则，先记录为待确认问题，不直接写入自动化断言。
- 自动化优先覆盖稳定、高风险、可重复执行且不依赖固定数据的场景。

## 需求到测试用例生成

```text
你是资深软件测试工程师。
请根据以下日程安排 CRUD 需求生成测试用例。

要求：
1. 覆盖正常流程、异常流程、边界值、安全场景和兼容性场景。
2. 输出字段：用例标题、前置条件、测试步骤、预期结果、优先级、是否适合自动化、建议测试层级。
3. 建议测试层级只能从以下选项中选择：后端单元测试、后端接口测试、前端单元测试、前端组件测试、Playwright API 测试、Playwright UI E2E、手工测试。
4. 标记哪些用例已经被现有自动化覆盖，哪些是新增建议。
5. 不要编造需求中没有定义的业务规则；如果发现规则不明确，请输出“待确认问题”。

需求内容：
{{requirements}}

现有测试覆盖：
{{existing_test_summary}}
```

## API 自动化补充建议

```text
你是 API 自动化测试工程师。
请根据以下接口契约和现有 Playwright API 测试，找出还值得补充的 API 自动化用例。

要求：
1. 优先选择稳定、可自动清理测试数据、不会依赖执行顺序的场景。
2. 每条建议说明覆盖的业务规则编号，例如 R001、R002。
3. 排除已经由后端单元测试或 UI E2E 充分覆盖、且 API 层收益较低的重复场景。
4. 对标题超长、跨天日程、特殊字符、非法枚举、非法 ID、非法日期筛选等场景给出建议。
5. 输出表格：用例标题、接口、输入要点、预期结果、优先级、是否本轮落地。

接口契约：
{{api_contract}}

现有 API 测试：
{{api_tests}}
```

## UI E2E 风险用例筛选

```text
你是前端 E2E 测试负责人。
请基于日程安排 CRUD 页面和现有 E2E 用例，筛选值得加入 UI 自动化的用户路径。

要求：
1. 优先覆盖真实用户高频路径，不把所有字段组合都放进 E2E。
2. 标记哪些场景更适合前端组件测试或 API 测试。
3. 特别关注筛选、编辑、删除确认、服务端错误、网络失败、刷新后列表状态。
4. 每条用例说明：用户价值、失败风险、自动化稳定性、建议层级。

页面行为：
{{ui_behavior}}

现有 E2E 测试：
{{e2e_tests}}
```

## 人工评审清单

使用 AI 输出后，逐条检查：

- 是否对应明确需求或已确认的业务规则。
- 是否和现有测试重复。
- 是否能稳定自动化执行。
- 测试数据是否能隔离和清理。
- 失败时是否能定位到具体接口、页面或业务规则。
- 是否需要先修改需求文档再实现测试。

## 本轮人工结论

- 适合立即补充到 API 自动化：非法状态筛选、非法日期筛选、编辑不存在日程、非法 ID、特殊字符、跨天日程按开始日期筛选。
- 暂不自动化：标题超长。当前需求未定义最大长度，后端也未实现长度限制，先记录为待确认。
- 暂不扩展 UI E2E：当前已有核心新增、筛选、编辑、删除和表单校验路径。网络失败、服务端错误更适合先放在组件测试或后续异常注入测试中。
